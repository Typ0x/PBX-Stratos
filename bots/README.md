# @pbx/bots

Fastify control plane for PBX trading bots. Serves the operations
dashboard at `/dashboard` and the underlying REST API. Runs identically
in production (Render) and locally — same single Node process, same
on-disk state layout under `$BOTS_DATA_DIR`.

> **Running inside PBX-Stratos?** `pm2.config.cjs` sets
> `STRATOS_BOTS_DATA_DIR=<repo>/runtime/bots` and exports it as
> `BOTS_DATA_DIR` so the upstream `~/.pbx-bots` defaults documented
> below are overridden to point at the repo-local `runtime/bots/`
> tree (Layer 3 in the three-layer model — see [`../CLAUDE.md`](../CLAUDE.md)).
> The text below documents standalone @pbx/bots usage; inside PBX-Stratos,
> substitute `runtime/bots/` for every `~/.pbx-bots/` mention.

## Quickstart (local)

```bash
export HELIUS_MAINNET_URL=...           # required; get one at helius.dev
npm --workspace bots run server
# → open http://localhost:8787/dashboard
```

That's it. On first boot the server will:

- Create `~/.pbx-bots/` (mode `0700`)
- Generate `BOT_API_TOKEN` and `BOT_MASTER_KEY`, write them to
  `~/.pbx-bots/local.env` (mode `0600`), reuse them on every subsequent boot
- Bind to `127.0.0.1:8787` so the autogenerated token can't be reached
  from your LAN
- The dashboard reads the token from `/api/local-token` (loopback-only
  endpoint) so you never see an auth overlay

> **Back up `~/.pbx-bots/local.env` before funding any wallet.** Losing
> `BOT_MASTER_KEY` orphans every encrypted wallet on disk — same security
> model as a hardware wallet seed phrase. The server refuses to start
> with a mismatched key (canary check).

## Environment variables

| Var | Default | Required? |
|---|---|---|
| `HELIUS_MAINNET_URL` | — | **yes** |
| `BOTS_DATA_DIR` | `~/.pbx-bots` (local) / `/var/data/pbx-bots` (Render) | no |
| `HOST` | `127.0.0.1` (local) / `0.0.0.0` (Render via `render.yaml`) | no |
| `PORT` | `8787` | no |
| `BOT_API_TOKEN` | autogenerated | no (locally) / yes (prod) |
| `BOT_MASTER_KEY` | autogenerated | no (locally) / yes (prod) |
| `DATABASE_URL` | — | optional; only used to backfill PM2.5 history at boot |

If `BOT_API_TOKEN` and `BOT_MASTER_KEY` are both set, the server uses
them verbatim. If neither is set, autogen fires. **Setting only one** is
a configuration error and the server refuses to start (would cause a
canary mismatch on next boot and orphan funds).

## Paper trading setup

The paper-trading subsystem (`factory paper-deploy`, the auto-promote
hook, the dashboard's "Deploy as paper bot" button) requires two
environment variables to be set BEFORE you launch the server or run
the factory CLI.

The server's bootstrap autogenerates `BOT_MASTER_KEY` into
`~/.pbx-bots/local.env` on first boot (see Quickstart above), but the
**factory CLI does NOT source `local.env`** — when you invoke
`npx tsx scripts/backtest/factory/cli.ts …` directly, env vars must
already be exported in your shell.

### Required env vars

| var              | shape                       | purpose                                                                       |
|------------------|-----------------------------|-------------------------------------------------------------------------------|
| `BOT_MASTER_KEY` | hex string, ≥ 32 chars      | Encrypts wallet keypairs at rest (scrypt-derived AES-256-GCM). Treat as a secret. |
| `BOTS_DATA_DIR`  | absolute writeable path     | Where per-bot state (keypair, NAV history, trade log, provenance) is written.    |

Defined / consumed in `src/server/secrets.ts`, `src/server/index.ts`,
and `scripts/backtest/factory/{paper-deploy,observer,correlation,
evolve/auto-promote}.ts`. Without them:

- `factory paper-deploy` throws `BOT_MASTER_KEY env not set or too
  short (min 32 chars)` from `secrets.ts` and exits non-zero.
- The auto-promote hook (PR #76) returns
  `{ promoted: false, reason: 'env-missing' }` — a clear log line and
  a silent skip; the evolve loop keeps running.

### One-time setup

Generate a random master key and pick a data directory:

```bash
echo "export BOT_MASTER_KEY=$(openssl rand -hex 32)" >> ~/.zshrc
echo "export BOTS_DATA_DIR=$HOME/.pbx-bots"          >> ~/.zshrc
source ~/.zshrc
mkdir -p "$BOTS_DATA_DIR"
```

(Substitute `~/.bashrc` for bash shells.)

If you've already run the server locally once, `~/.pbx-bots/local.env`
already contains a `BOT_MASTER_KEY` line — re-use that exact value so
the factory CLI and the server share a key (mismatched keys would
orphan any wallets the server has already encrypted):

```bash
grep BOT_MASTER_KEY ~/.pbx-bots/local.env >> ~/.zshrc
echo "export BOTS_DATA_DIR=$HOME/.pbx-bots" >> ~/.zshrc
# add `export ` in front of the grepped line, then `source ~/.zshrc`
```

### Verify

```bash
cd bots
npx tsx scripts/backtest/factory/cli.ts paper-deploy --top 0
```

You should see no `BOT_MASTER_KEY env not set` error and a clean
`0 paper bot(s) launched` (or the equivalent no-op log). If you see
`BOT_MASTER_KEY env not set or too short` or `BOTS_DATA_DIR` complaints,
the env vars didn't propagate — re-source the rc file in the shell
you're running the CLI from.

### What unlocks once these are set

- `factory paper-deploy --top N --capital-usdc M` actually launches paper bots
- `factory observer` starts producing rows from
  `$BOTS_DATA_DIR/state/nav-history.jsonl` + per-bot provenance
- `factory allocate` (and the continuous allocator daemon) operates on
  real paper data
- `factory correlation` correlates real running bots
- The auto-promote hook (PR #76) deploys leaderboard winners
  automatically when new-bests fire (capped at 50 USDC, paper-only)
- The dashboard's "Deploy as paper bot" button writes a provenance
  file under `$BOTS_DATA_DIR/provenance/` that the observer joins
  against NAV + trade history

Until then the loop fails-safe: every paper-deploy call silently skips
with a warning, so the evolve loop keeps producing backtest experiments
without ever touching live or paper trading.

### Security note

`BOT_MASTER_KEY` is the single secret that protects all on-disk wallet
keypairs (canary-checked at server boot — see `src/server/secrets.ts`).
If you rotate it, encrypted bots become unreadable — back up
`$BOTS_DATA_DIR` before changing it. Treat it like a `.env` secret: do
NOT commit it, do NOT paste it in chats, do NOT export it from
tmuxinator / shell-rc files that get checked in. Losing it orphans
funds the same way losing a hardware-wallet seed does.

## LAN access

The default binds to loopback. To expose the server to your LAN you
**must pin secrets explicitly** — the server refuses to start with
autogenerated secrets bound to a non-loopback host:

```bash
export BOT_API_TOKEN=$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")
export BOT_MASTER_KEY=$(node -e "console.log(require('crypto').randomBytes(32).toString('hex'))")
export HOST=0.0.0.0
export HELIUS_MAINNET_URL=...
npm --workspace bots run server
```

The dashboard's auth overlay appears for any non-loopback origin, and
the auth-bypass logic strictly checks `req.socket.remoteAddress` (not
`req.ip`), so a reverse-proxy / `trustProxy` misconfig cannot turn the
loopback bypass into a remote auth bypass.

## Production (Render)

Deploys via `render.yaml` in this directory. All four env vars
(`BOT_API_TOKEN`, `BOT_MASTER_KEY`, `HELIUS_MAINNET_URL`, `DATABASE_URL`)
must be set in the Render dashboard before first deploy. `HOST` and
`BOTS_DATA_DIR` are pinned in `render.yaml`.

## Layout

```
├── render.yaml                 # Render service definition
├── src/
│   ├── server/

│   │   ├── dashboard.html      # the operations UI
│   │   ├── store.ts            # persistent state on disk
│   │   ├── secrets.ts          # AES-256-GCM keypair encryption
│   │   ├── orchestrator.ts     # bot lifecycle
│   │   └── nav-snapshotter.ts  # NAV history for the chart
│   ├── strategies/             # region_arb_*, pm25_*, etc.
│   ├── core/pm25_history.ts    # PM2.5 history buffer + percentile/z-score
│   └── cli.ts                  # local-CLI entrypoint (separate from server)
└── package.json
```
